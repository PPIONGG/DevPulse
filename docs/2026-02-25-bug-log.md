# Bug Log — 2026-02-25

Phase 1 project initialization. Summary of all bugs and issues encountered during development.

---

## Bug #1: Next.js build fails — Supabase env vars missing during prerender

**Symptom:** `npm run build` fails with `@supabase/ssr: Your project's URL and API key are required to create a Supabase client!`

**Cause:** Authenticated pages under `(app)/` were being statically prerendered at build time, which triggered the Supabase middleware without env vars.

**Fix:** Added `export const dynamic = "force-dynamic"` to `frontend/src/app/(app)/layout.tsx` to skip static prerendering for authenticated routes.

---

## Bug #2: TypeScript compile error — duplicate `value` attribute

**Symptom:** Build fails with `JSX elements cannot have multiple attributes with the same name` on the login password input.

**Cause:** Copy-paste error — the password `<Input>` had both `value={email}` and `value={password}`.

**Fix:** Removed the duplicate `value={email}` line, kept only `value={password}`.

---

## Bug #3: No email/password registration

**Symptom:** Login page only had GitHub OAuth button. No way to register with email and password.

**Cause:** Initial implementation only included GitHub OAuth per the original plan.

**Fix:** Rebuilt login page with Tabs (Login / Register) for email+password auth alongside the GitHub OAuth button. Installed shadcn `input`, `label`, `card`, `tabs` components.

---

## Bug #4: "Check your email for confirmation link" after registration

**Symptom:** After registering with email/password, user sees confirmation message but can't log in. No email arrives.

**Cause:** Supabase has **email confirmation** enabled by default. Without SMTP configured, confirmation emails don't send.

**Fix:** Disabled "Confirm email" in **Supabase Dashboard > Authentication > Settings** for development.

---

## Bug #5: SQL error — `relation "profiles" already exists`

**Symptom:** Running the profiles table creation SQL returns `ERROR: 42P07: relation "profiles" already exists`.

**Cause:** The `profiles` table had already been created previously (possibly from Supabase quickstart).

**Fix:** Skipped table creation. Queried `information_schema.columns` to verify existing schema matched, then only added the missing trigger and RLS policies.

---

## Bug #6: Profiles table missing trigger and RLS policies

**Symptom:** Registration works but no profile row is created. Users can't query their own profile.

**Cause:** The `profiles` table existed but had no auto-creation trigger (`on_auth_user_created`) and no RLS policies.

**Fix:** Added:
- RLS policies for SELECT and UPDATE (own profile only)
- Trigger function `handle_new_user()` that auto-inserts a profile row on signup
- Backfill SQL for existing users: `INSERT INTO profiles ... FROM auth.users WHERE id NOT IN (SELECT id FROM profiles)`

---

## Bug #7: Sidebar user menu stuck in loading state

**Symptom:** Bottom-left of sidebar shows skeleton loading animation indefinitely. No user name, avatar, or logout button.

**Cause:** `useEffect` in `AuthProvider` had `[supabase.auth]` as dependency. Since `createClient()` was called in the component body, `supabase.auth` could produce a new reference on re-renders, causing the effect to re-run in a loop. Additionally, `fetchProfile` had no error handling — if it failed, `setLoading(false)` was never reached.

**Fix:**
- Wrapped `createClient()` in `useMemo` so it's stable across renders
- Changed dependency array to `[supabase, fetchProfile]`
- Wrapped `fetchProfile` in `useCallback`
- Added `try/catch/finally` to ensure `setLoading(false)` always runs

---

## Bug #8: Settings page — Display Name field empty despite having a name

**Symptom:** User has a saved display name, but the Settings input field shows empty placeholder "Your name".

**Cause:** `useState(profile?.display_name ?? "")` captures the value at mount time. Since `profile` is `null` during initial render (still loading from auth provider), the state initializes to `""` and never updates.

**Fix:** Added `useEffect` to sync state when profile loads:
```tsx
useEffect(() => {
  if (profile) {
    setDisplayName(profile.display_name ?? "");
    setAvatarUrl(profile.avatar_url ?? "");
  }
}, [profile]);
```

---

## Bug #9: Avatar upload — 404 "Bucket not found"

**Symptom:** Uploading avatar returns `{"statusCode":"404","error":"Bucket not found","message":"Bucket not found"}`.

**Cause:** The `avatars` storage bucket was never created in Supabase. Creating via Dashboard didn't work for the user.

**Fix:** Created bucket via SQL Editor:
```sql
INSERT INTO storage.buckets (id, name, public) VALUES ('avatars', 'avatars', true);
```

---

## Bug #10: Avatar image distorted / squeezed

**Symptom:** Uploaded portrait photo appears squeezed/distorted in the circular avatar.

**Cause:** Non-square images were uploaded as-is and forced into a circle, causing distortion.

**Fix:** Added `react-image-crop` library. Users now select a file, crop it to a square (circular preview) interactively, then upload. The cropped image is resized to 256x256 via canvas before upload.

---

## Bug #11: Settings page — Sidebar doesn't update after saving

**Symptom:** After changing display name or avatar in Settings and clicking Save, the sidebar still shows old data until page refresh.

**Cause:** `handleSave` and `handleAvatarUpload` updated the database but didn't refresh the auth context.

**Fix:** Added `refreshProfile()` to auth provider and called it after successful save/upload:
```tsx
await refreshProfile();
```

---

## Bug #12: Git push — Permission denied (publickey)

**Symptom:** `git push -u origin main` fails with `Permission denied (publickey)`.

**Cause:** SSH key not configured for GitHub. HTTPS also failed because GitHub no longer accepts password authentication.

**Fix:** Installed `gh` CLI (`brew install gh`), authenticated with `gh auth login`, then pushed successfully.
